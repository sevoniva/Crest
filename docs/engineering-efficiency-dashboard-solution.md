# 研发效能数据大屏方案

本文根据《研发效能体系-0528》材料和 Crest 当前内置研发效能大屏整理，作为后续接入真实研发管理数据时的指标口径和数据建模依据。

## 1. 设计原则

这套大屏不是“把数据凑成图”，而是把研发管理问题拆成可解释、可追溯、可下钻的度量体系。

核心假设如下：

1. 研发效能度量以团队、系统、价值流、项目组合为主，个人层面只用于工作台、负载透明和自我改进，不用于单指标绩效排名。
2. 工时、代码行、提交数只能作为工程活动信号，不能单独代表个人生产力。
3. 所有指标必须有来源系统、统计粒度、时间口径、分子分母、过滤条件、责任人和明细下钻路径。
4. 需求必须有状态流转历史，否则无法科学计算等待时长、阶段时效、累积流图、WIP 和瓶颈。
5. 演示数据可以模拟，但必须按“真实系统可采集字段”组织，不能只造结果表。

## 2. 对齐的业内框架和规范

| 框架/规范 | 用在本方案中的位置 | 采用原因 |
| --- | --- | --- |
| DORA software delivery performance metrics | 工程交付能力、部署频率、变更前置时间、变更失败率、失败部署恢复时间、部署返工率 | 衡量软件交付吞吐与稳定性，适合系统/服务级持续改进 |
| Kanban Guide Flow Metrics | 需求流动、WIP、Throughput、Work Item Age、Cycle Time、SLE、累积流图 | 解释“需求为什么堵、堵在哪里、堵多久”，和 PPT 中累积流图高度一致 |
| SPACE developer productivity framework | 防止把代码行/工时/提交数误当生产力；平衡满意度、绩效、活动、协作、效率流动 | 研发效能是多维系统，不应单指标化 |
| ISO/IEC/IEEE 29148 Requirements Engineering | 需求生命周期、需求信息项、需求过程和追踪 | 支撑需求从提出、分析、就绪、开发、验收、发布的可追溯 |
| ISO/IEC 25010 Product Quality Model | 质量、缺陷、可靠性、可维护性等质量类指标 | 支撑质量与稳定性大屏，不只看缺陷数量 |
| 精益价值流/VSM 思路 | 增值、运营性增值、不增值活动分类；瓶颈和等待消除 | 对应 PPT 中“加速增值活动，消除不增值和缩短运营性增值活动” |

参考链接：

- DORA: https://dora.dev/guides/dora-metrics/
- Kanban Guide: https://kanbanguides.org/the-kanban-guide/2025.5/
- SPACE: https://queue.acm.org/detail.cfm?id=3454124
- ISO/IEC/IEEE 29148: https://www.iso.org/standard/72089.html
- ISO/IEC 25010: https://www.iso.org/standard/78176.html

## 3. 大屏总体分层

建议做 5 张主大屏，外加一个通用下钻明细页。

### 3.1 研发经营总览大屏

面向：行领导、总经理室、科技管理层。

回答的问题：

- 全行研发投入是否换来了稳定、可预测的交付？
- 哪些部门/系统/项目组合存在容量、质量、进度风险？
- 预算、需求、交付、质量、运行是否在同一张图上闭环？

核心图表：

| 图表 | 指标 | 来源 | 下钻 |
| --- | --- | --- | --- |
| 四象限总览 | 响应力、产能、质量、满意度 | 需求系统、CI/CD、缺陷、ITSM、满意度问卷 | 行级 -> 部门 -> 科室 -> 系统/项目 |
| 需求吞吐趋势 | 每周/月上线需求数 | 需求系统、发布系统 | 月份 -> 部门 -> 系统 -> 需求清单 |
| 计划投产完成率 | 按期上线数 / 计划上线数 | 需求系统、发布计划 | 部门 -> 批次 -> 需求 |
| 交付时长分布 | 需求从录入到投产的 P50/P85/P95 | 需求状态流转历史 | 部门 -> 系统 -> 需求 |
| DORA 指标 | 部署频率、变更前置时间、变更失败率、失败部署恢复时间、部署返工率 | Git、CI/CD、发布、ITSM | 系统 -> 部署批次 -> 变更/提交 |
| 预算执行 | 实际成本 / 核定预算，资本化/费用化拆分 | 预算、采购、工时、外包合同 | 部门 -> 项目 -> 费用明细 |
| 高风险项目列表 | 延期、超预算、质量红线、容量不足 | 项目、需求、缺陷、工时 | 项目 -> 里程碑 -> 风险项 |

### 3.2 需求流动与交付效率大屏

面向：研发管理、项目经理、需求经理、产品负责人。

回答的问题：

- 需求是入口堵、开发堵、测试堵，还是发布堵？
- 哪个阶段等待最长？哪个团队 WIP 过高？
- 需求吞吐是在改善，还是只是堆积变大？

核心图表：

| 图表 | 指标 | 来源 | 下钻 |
| --- | --- | --- | --- |
| 部署需求累积流图 | 每日各阶段需求存量 | 需求状态事件表/每日快照 | 阶段 -> 部门 -> 系统/项目 -> 优先级 -> 需求 |
| 需求漏斗 | 意向需求、待受理、待分析、待排期、待开发、待测试、待上线 | 需求系统 | 阶段 -> 需求类型 -> 需求 |
| 阶段时效箱线图 | 各阶段停留时长 P50/P85/P95 | 需求状态事件表 | 阶段 -> 部门 -> 需求 |
| WIP 与 SLE 超期 | 未完成需求数、Work Item Age、SLE 超期数 | 状态事件表、SLE 配置 | 团队 -> 需求 |
| 阻塞透明 | 阻塞次数、阻塞天数、阻塞原因 TOP | 看板/需求系统阻塞标签 | 原因 -> 责任团队 -> 需求 |
| 吞吐量控制图 | 每周完成需求数、移动均值、异常点 | 需求系统 | 周 -> 部门 -> 需求 |

累积流图是本方案的关键图，不是普通面积图。它的科学口径见第 7 节。

### 3.3 人力容量与供应商经营大屏

面向：科技管理、人力资源、供应商管理、部门负责人。

回答的问题：

- 当前人力容量是否匹配需求池？
- 角色结构是否健康？外包比例、供应商质量和交付是否可控？
- 哪些团队是能力短板，哪些团队是需求承压？

核心图表：

| 图表 | 指标 | 来源 | 下钻 |
| --- | --- | --- | --- |
| 角色分布透视 | 人数/FTE 按角色、部门、内外部、供应商分布 | HR、组织、供应商主数据 | 部门 -> 角色 -> 人员/供应商 |
| 容量需求比 | 可用容量工时 / 需求预测工时 | HR 日历、工时、需求估算 | 部门 -> 角色 -> 项目 |
| 外包人员等级分布 | 外包人数按等级、类型、供应商 | 供应商合同、HR | 供应商 -> 等级 -> 人员 |
| 供应商交付质量 | SLA 达成率、按期交付率、缺陷逃逸率、返工工时 | 供应商、需求、缺陷、工时 | 供应商 -> 项目 -> 需求 |
| 资源冲突 | 同一人并行需求数、超负载天数 | 任务计划、工时、排期 | 人员 -> 日期 -> 任务 |
| 技能与需求匹配 | 需求技能标签 vs 人员技能/FTE | 技能库、需求标签 | 技能 -> 团队 -> 人员 |

### 3.4 工程活动与代码效能大屏

面向：研发负责人、架构、DevOps、工程效率团队。

回答的问题：

- 工程活动是否健康流动？
- 代码提交、评审、构建、部署哪个环节慢？
- 工时和代码行如何分类解释，而不是误用为绩效？

核心图表：

| 图表 | 指标 | 来源 | 下钻 |
| --- | --- | --- | --- |
| 工时分类 | 增值、运营性增值、不增值/等待、返工 | 工时系统、任务类型映射 | 部门 -> 项目 -> 人员 -> 工时明细 |
| 代码变更趋势 | 新增行、删除行、净行、变更文件数 | Git | 系统 -> 仓库 -> PR/commit |
| 代码返工/变更 churn | 14/30 天内被修改或删除的新增代码占比 | Git | 仓库 -> PR -> 文件 |
| PR 流动 | PR 创建到合并时长、评审等待时长、返修次数 | Git 平台 | 仓库 -> PR -> review |
| CI 健康 | 构建成功率、平均构建耗时、失败原因 | CI | 系统 -> pipeline -> build |
| 发布节奏 | 部署频率、批量大小、紧急发布占比 | CD、发布系统 | 系统 -> 发布批次 |

代码行只展示“活动强度、变更规模、返工信号”，不做“代码行越多越好”的评价。

### 3.5 质量稳定性与风险预警大屏

面向：质量管理、运维、研发负责人、系统负责人。

回答的问题：

- 上线质量是否稳定？
- 缺陷是在测试阶段暴露，还是逃逸到生产？
- 哪些系统/需求/供应商带来最大运行风险？

核心图表：

| 图表 | 指标 | 来源 | 下钻 |
| --- | --- | --- | --- |
| 测试缺陷密度 | 测试缺陷数 / 迭代内需求数 | 缺陷系统、需求系统 | 系统 -> 迭代 -> 缺陷 |
| 生产问题密度 | 上线 30 天内生产问题数 / 上线需求数 | ITSM、发布、需求 | 系统 -> 发布 -> 问题 |
| 缺陷逃逸率 | 生产缺陷 / 全部缺陷 | 缺陷、ITSM | 系统 -> 需求 -> 缺陷 |
| 严重问题 TOP | P1/P2 问题数、恢复时长 | ITSM、监控 | 系统 -> 事件 |
| 变更失败率 | 失败部署/需人工干预部署 / 生产部署数 | CD、ITSM | 系统 -> 部署 |
| 风险预警 | 进度偏差、容量不足、SLE 超期、质量超阈值 | 项目、需求、工时、缺陷 | 预警 -> 明细对象 |

## 4. 通用下钻规则

所有聚合图必须能下钻到可解释明细，不允许只有总数。

推荐统一下钻链路：

1. 组织链路：全行 -> 部门 -> 科室 -> 团队 -> 人员/供应商。
2. 业务链路：需求域 -> 系统 -> 项目/批次 -> 需求 -> 任务。
3. 工程链路：系统 -> 仓库 -> PR/MR -> commit -> 文件。
4. 质量链路：系统 -> 发布批次 -> 缺陷/生产问题 -> 根因/责任闭环。
5. 财务链路：预算科目 -> 部门 -> 项目 -> 合同/工时/费用明细。

每个图表至少配置：

- 统计周期过滤：日、周、月、迭代、年度。
- 组织过滤：部门、科室、团队。
- 系统/项目过滤：系统、项目、批次。
- 需求过滤：类型、优先级、来源、规模、状态。
- 明细表：展示参与计算的原始对象 ID、状态、时间、负责人、来源系统更新时间。

## 6. 核心指标口径

### 6.1 需求与流动

| 指标 | 公式/口径 | 说明 |
| --- | --- | --- |
| 需求吞吐量 | 统计周期内完成并上线的需求数 | Kanban 中 throughput 是单位时间完成工作项的准确计数，不建议再除一次周期后混淆口径；如需“月均吞吐”，再做平均 |
| 计划投产完成率 | 按期上线需求数 / 计划上线需求数 * 100% | PPT 中出现“计划上线数 / 按时上线数”的写法，应修正，否则会超过 100% 且方向相反 |
| 需求交付时长 Lead Time | actual_go_live_at - created_at | 建议看 P50/P85/P95，不只看平均值 |
| 需求等待时长 | ready_for_dev_at - created_at，或按组织定义为 dev_start_at - created_at | 必须在指标卡中说明“待开发”的状态定义 |
| 研发周期 Cycle Time | actual_go_live_at - dev_start_at，或 done_at - start_at | started/finished 必须在 DoW 中固定 |
| 阶段停留时长 | stage_left_at - stage_entered_at | 针对已离开阶段的需求 |
| 当前阶段年龄 Work Item Age | 当前日期 - current_stage_entered_at | 针对未完成需求 |
| WIP | started_at 不为空且 finished_at 为空的需求数 | 可按团队、阶段、系统拆分 |
| SLE 超期率 | 当前年龄超过 SLE 的未完成需求数 / 未完成需求数 | SLE 需基于历史周期，例如“85% 需求 10 天内完成” |

### 6.2 人力容量

| 指标 | 公式/口径 | 说明 |
| --- | --- | --- |
| 可用容量工时 | Σ FTE * 工作日标准工时 * 项目分配比例 * (1 - 请假/培训/运维预留比例) | 按角色、团队、月份计算 |
| 需求容量工时 | Σ 需求估算工时；无估算时用需求类型/复杂度的历史中位工时预测 | 不建议只用需求数量比较容量 |
| 容量需求比 | 可用容量工时 / 需求容量工时 * 100% | <90% 容量不足，90%-110% 相对均衡，>110% 需结合闲置或需求储备解释 |
| 并行负载 | 人员同周期活跃任务/需求数 | 用于识别上下文切换风险 |
| 角色缺口 | 需求所需角色工时 - 可用角色工时 | 按 BA/开发/测试/架构/运维/产品拆分 |

### 6.3 供应商

| 指标 | 公式/口径 | 说明 |
| --- | --- | --- |
| 供应商 SLA 达成率 | SLA 达标任务/需求数 / SLA 适用任务/需求数 * 100% | SLA 类型需区分响应、交付、修复 |
| 外包交付按期率 | 按期完成的外包参与需求数 / 外包参与需求总数 * 100% | 需按需求复杂度分层，避免供应商之间不公平比较 |
| 外包缺陷逃逸率 | 外包参与上线后 30 天生产问题数 / 外包参与上线需求数 | 用于质量风险，不直接等于责任归属 |
| 单位交付成本 | 外包成本 / 完成需求规模 | 需求规模优先用功能点/故事点/估算工时，否则只做粗略参考 |
| 返工工时占比 | 返工工时 / 总工时 * 100% | 结合需求变更原因分析 |

### 6.4 工时与代码

| 指标 | 公式/口径 | 说明 |
| --- | --- | --- |
| 工时分类 | 按 work_type 映射为增值、运营性增值、不增值/等待、返工 | 映射表要可配置、可审计 |
| 净代码行 | added_lines - deleted_lines | 排除生成代码、vendor、锁文件、格式化-only 提交 |
| 代码 churn | 14/30 天内被删除或修改的新增代码行 / 新增代码行 | 反映返工或需求不稳定，不直接归责个人 |
| PR 合并时长 | merged_at - opened_at | 拆成评审等待、返修、CI 等阶段更有解释力 |
| 评审等待时长 | first_review_at - opened_at | 可用于识别协作瓶颈 |

### 6.5 质量与稳定性

| 指标 | 公式/口径 | 说明 |
| --- | --- | --- |
| 测试缺陷密度 | 系统测试缺陷数 / 迭代内需求数 | 有规模数据时升级为 /功能点或/估算工时 |
| 生产问题密度 | 上线 30 天内生产问题数 / 迭代内上线需求数 | 和 PPT 口径一致 |
| 缺陷逃逸率 | 生产缺陷数 / 全部缺陷数 * 100% | 可按严重等级加权 |
| 变更失败率 | 需回滚/热修/人工干预的生产部署数 / 生产部署数 * 100% | DORA 口径 |
| 失败部署恢复时间 | recover_at - failure_detected_at | DORA 当前推荐指标之一 |
| 部署返工率 | 因生产事件触发的非计划部署数 / 部署总数 * 100% | DORA 当前五指标之一 |

## 7. 部署需求累积流图的科学口径

这张图必须加，而且要作为“需求流动与交付效率大屏”的中心图。

### 7.1 图表定义

图表类型：堆叠面积/堆叠折线图。

横轴：日期。

纵轴：需求数量。

堆叠维度：需求阶段。阶段名称来自指标字典或需求系统状态映射，不应写死在图表组件里。推荐标准顺序：

1. 需求提出
2. 受理澄清
3. 需求确认
4. 研发立项
5. 开发中
6. 测试验收
7. 发布上线

统计口径：

在每个快照日期，对每个需求取该日 23:59:59 的最新状态，每个需求只能落入一个阶段。已上线需求保留在“发布上线”阶段到观察窗口结束，用于观察累计交付产出。

计算逻辑：

```sql
-- 逻辑示意
stage_on_day =
  latest status_event
  where entered_at <= snapshot_date_end
    and (left_at is null or left_at > snapshot_date_end)

req_count =
  count(distinct requirement_id)
  group by snapshot_date, stage_name
```

### 7.2 如何读图

| 现象 | 解释 | 管理动作 |
| --- | --- | --- |
| 顶部总量持续抬升 | 总需求池/backlog 增大 | 检查入口准入、优先级和容量规划 |
| 某一颜色带持续变厚 | 该阶段需求堆积，可能是瓶颈 | 下钻阶段停留时长和阻塞原因 |
| 开发中面积过宽 | WIP 高、多任务切换风险 | 控制并行、拆小批量、设置 WIP 限制 |
| 测试验收或发布上线前面积变厚 | 测试、发布窗口或上线审批排队 | 增加发布频次、优化变更审批 |
| 发布上线面积变厚 | 交付产出增加 | 同时观察质量和生产问题密度，避免只追吞吐 |

### 7.3 下钻路径

点击某天某阶段面积：

1. 日期 + 阶段需求清单。
2. 部门/科室分布。
3. 系统/项目分布。
4. 优先级、需求类型、来源部门。
5. 单个需求状态流转时间线、阻塞记录、负责人、相关任务/PR/缺陷。

### 7.4 平台实现建议

当前 Crest 前端已有“堆叠折线图/area-stack”能力，可优先用内置图表实现。

如后续需要以下能力，再开发通用组件 `CumulativeFlowChart`，不能做一次性页面组件：

- 图内直接展示阶段 SLE 超期红点。
- 支持点击面积弹出需求清单。
- 支持叠加吞吐量折线、WIP 控制线。
- 支持阻塞事件标注。
- 支持阶段颜色和顺序由指标字典配置。

## 8. 推荐数据模型

### 8.1 维表

- `dim_org`：组织、部门、科室、团队。
- `dim_person`：人员、角色、职级、雇佣类型、供应商、FTE。
- `dim_vendor`：供应商、合同、服务类型、SLA。
- `dim_system`：系统、业务域、负责人、等级。
- `dim_project`：项目、批次、预算、生命周期。
- `dim_requirement`：需求主数据。
- `dim_calendar`：日期、工作日、节假日、迭代、月、季度。
- `dim_metric_definition`：指标定义、来源、公式、口径、负责人。

### 8.2 事实表

- `fact_requirement_status_event`：需求状态流转事件。
- `fact_requirement_snapshot_daily`：需求每日状态快照，用于累积流和 WIP。
- `fact_requirement_delivery`：需求交付结果。
- `fact_project_milestone`：项目里程碑计划与实际。
- `fact_capacity_daily`：人员/角色可用容量。
- `fact_worklog_daily`：审批后工时。
- `fact_code_change`：commit/PR 代码变更。
- `fact_pr_review`：评审与合并过程。
- `fact_ci_build`：构建流水线。
- `fact_deployment`：部署发布。
- `fact_defect`：测试缺陷。
- `fact_incident`：生产事件/问题。
- `fact_budget_execution`：预算与成本执行。
- `bridge_work_item_code`：需求/任务与 PR、commit、部署的映射。

## 9. 数据质量与治理要求

每个指标上线前必须通过以下检查：

1. 唯一性：需求、人员、系统、供应商、PR、部署、缺陷有稳定 ID。
2. 完整性：关键时间点不为空，无法补齐的要有缺失率。
3. 一致性：状态流转顺序符合流程，不允许上线时间早于创建时间。
4. 可追溯：聚合数能还原到明细对象。
5. 新鲜度：每个数据源标注刷新频率和最后同步时间。
6. 权限：个人、供应商、成本数据分级授权。
7. 口径版本：指标公式变更要留版本，不覆盖历史口径。

## 10. 当前平台落地计划

### 阶段 1：标准化演示数据 v2

目标：在本地 Crest 上做出可演示的 5 张大屏，数据虽然是模拟的，但结构与真实接入一致。

工作内容：

1. 增加指标定义表和数据来源说明表。
2. 将现有造数从“结果视图”升级为“原始事实表 + 汇总视图”。
3. 补齐需求状态事件和每日快照，重做部署需求累积流图。
4. 每张图配置下钻字段和明细表。
5. 页面上增加“指标口径/来源说明”入口。

### 阶段 2：真实系统接入

目标：替换模拟数据为真实系统数据。

接入优先级：

1. 需求系统和状态流转。
2. HR/组织/供应商主数据。
3. 工时系统。
4. Git 与 PR。
5. CI/CD 与发布。
6. 缺陷与 ITSM。
7. 预算/合同。
8. 满意度问卷。

### 阶段 3：治理闭环

目标：从“看见数据”变为“指导改进”。

工作内容：

1. 设置 SLE、WIP、缺陷密度、生产问题密度等阈值。
2. 对风险图表增加预警清单和责任闭环。
3. 复盘报表自动生成：周期、瓶颈、根因、改进动作、验证指标。
4. 对管理层输出月报，对团队输出改进看板，对个人输出工作台。

## 11. 不建议做的事

1. 不把代码行、工时、提交数作为个人绩效排名。
2. 不只用平均值描述周期，必须补 P50/P85/P95。
3. 不把不同复杂度、不同系统等级的团队做简单横向排名。
4. 不用没有状态历史的数据计算阶段时效。
5. 不在没有需求-PR-部署-缺陷链路时强行计算 DORA。
6. 不做只有截图效果、没有明细下钻和指标口径的“大屏样子货”。
